Ось детальний **Тест-план для E2E та Accessibility (A11y) тестування** вашого застосунку з використанням `Playwright` та `@axe-core/playwright`.

### 1. Мета тестування
Забезпечити коректну роботу ключових користувацьких сценаріїв (End-to-End) та повну відповідність застосунку стандартам доступності (WCAG не менше рівня AA) на всіх етапах взаємодії, враховуючи динамічні зміни DOM (додавання завдань, контактів, помилки валідації).

### 2. Інструменти
* **E2E Фреймворк:** Playwright (`@playwright/test`)
* **Accessibility рушій:** `@axe-core/playwright`
* **Браузери:** Chromium, Firefox, WebKit (десктопні та мобільні роздільні здатності).
* **Теми:** Світла та Темна (емуляція через Playwright `colorScheme`).

---

### 3. Область тестування (Scope) та Сценарії (Test Cases)

#### 3.1. Загальні перевірки (Layout & Navigation)
* **TC-01: Навігація між сторінками**
  * *Кроки:* Відкрити головну сторінку -> Клікнути на "Todo" -> Клікнути на "Address Book".
  * *E2E Очікування:* URL змінюється коректно, контент сторінки рендериться.
  * *A11y Очікування:* Жодних порушень контрастності, навігаційні посилання мають правильні ролі (aria-current, nav). Відсутність порушень Axe на кожній сторінці під час завантаження.
* **TC-02: Підтримка клавіатури (Focus Management)**
  * *Кроки:* Пересуватися за допомогою клавіші `Tab` по меню.
  * *E2E/A11y Очікування:* Кожен інтерактивний елемент отримує видимий фокус, порядок табуляції логічний (зліва направо, зверху вниз).

#### 3.2. Сторінка Todo List (`/todo`)
* **TC-03: Додавання нового завдання**
  * *Кроки:* Ввести текст у поле вводу -> Натиснути "Add" (або Enter).
  * *A11y Очікування:* Базовий скан сторінки перед дією не має помилок. Скан *після* появи нового елемента в списку також не містить помилок (новостворені елементи мають правильну структуру списку `<ul>`/`<li>`).
* **TC-04: Зміна статусу завдання (Complete/Undo)**
  * *Кроки:* Клікнути по завданню (або чекбоксу).
  * *A11y Очікування:* Стан зміни доноситься до скрінрідерів (наприклад, через `aria-checked` або зміну стилів з достатнім контрастом).
* **TC-05: Видалення завдання**
  * *Кроки:* Натиснути кнопку видалення біля завдання.
  * *A11y Очікування:* Кнопка видалення має доступне ім'я (наприклад, `aria-label="Delete task"`), фокус не "губиться" після видалення (повертається на логічний наступний елемент).

#### 3.3. Сторінка Address Book (`/address-book`)
* **TC-06: Валідація форми (Порожня/Некоректна форма)**
  * *Кроки:* Відправити порожню форму або форму з невалідним email.
  * *A11y Очікування:* Скан Axe після виклику помилок. Тексти помилок пов'язані з відповідними полями вводу (через `aria-describedby` або `aria-errormessage`), поля мають `aria-invalid="true"`.
* **TC-07: Успішне додавання контакту**
  * *Кроки:* Заповнити всі поля валідними даними -> Натиснути "Save".
  * *E2E Очікування:* Контакт з'являється у списку.
  * *A11y Очікування:* Нова картка контакту доступна, всі іконки та текст мають достатній контраст і до них можна дістатися клавіатурою.

### 4. Стратегія впровадження (Implementation Strategy)

**4.1. Встановлення**
Для реалізації плану вам потрібно буде виконати:
```bash
npm init playwright@latest
npm install -D @axe-core/playwright
```

**4.2. Інтеграція Axe з Playwright (Приклад)**
Базовий підхід полягає в тому, щоб запускати перевірку доступності в критичних станах DOM:

```typescript
// tests/todo.spec.ts
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test.describe('Todo Page', () => {
  test('should have no automatically detectable accessibility issues', async ({ page }) => {
    await page.goto('/todo');

    // 1. Перевірка статичної сторінки
    let accessibilityScanResults = await new AxeBuilder({ page }).analyze();
    expect(accessibilityScanResults.violations).toEqual([]);

    // 2. Взаємодія з елементами (додавання)
    await page.getByPlaceholder('What needs to be done?').fill('Learn Playwright');
    await page.getByRole('button', { name: 'Add' }).click();

    // 3. Знову перевіряємо структуру після мутації DOM
    accessibilityScanResults = await new AxeBuilder({ page }).analyze();
    expect(accessibilityScanResults.violations).toEqual([]);
  });
});
```

**4.3. Тестування тем**
Playwright дозволяє легко перевизначати тему, що покриває вимогу з попереднього питання.
```typescript
test.use({ colorScheme: 'dark' }); // Можна передавати глобально в config або на рівні тесту
test('Dark mode a11y check', async ({ page }) => {
  await page.goto('/');
  const results = await new AxeBuilder({ page }).analyze();
  expect(results.violations).toEqual([]);
});
```

### 5. Інтеграція з CI/CD (GitHub Actions)
Ці E2E тести можна додати до існуючого a11y-check.yml або створити окремий workflow (`e2e.yml`). Playwright автоматично запускатиме `vite preview` перед тестами, якщо ви налаштуєте параметр `webServer` у файлі `playwright.config.ts`. Звіт щодо виявлених порушень Axe (який можна експортувати в HTML через `axe-html-reporter`) або візуальні репорти Playwright можна прикріплювати як артефакти в GitHub-артефакти.
